## 核心能力展示区（三栏并列）

### 能力 1：架构驱动 + 模块化自动开发 + 持续验证

| 元素 | 内容 |
|------|------|
| **能力标题** | 架构驱动：人画蓝图，AI 按模块拓扑顺序开发 + 自测 |
| **价值描述** | 架构设计是人的核心贡献。AI 按依赖关系逐个实现模块，每完成一个模块自动生成测试并跑到通过，才进入下一个模块。**编译失败 → AI 自行修复 → 再编译 → 直到通过**，人不介入。 |
| **提示词示例** | "这是架构设计：Ballista 扩展 7 层管线（proto → logical → physical → DataFrame → planner → codec → optimizer），按这个依赖顺序，从底层 proto 开始逐个实现。每个模块完成后写单元测试并跑到通过。" |
| **AI 自动执行** | ① proto 定义 + build.rs tonic 代码生成 → 编译通过 → ② `UserDefinedLogicalNodeCore` 实现 → 编译通过 → ③ `ExecutionPlan` 实现（含 C ABI 生命周期 5 个阶段）→ 编译通过 → ④ `DataFrameExt` API → 编译通过 → ⑤ `ExtensionPlanner` → 编译通过 → ⑥ `Codec` 序列化层 → 编译通过 → ⑦ `EnforceDistributeBy` 物理优化规则 → 编译 + 测试全部通过 |
| **关键差异** | 不是"一句话生成全部代码然后修编译错误到天黑"。**AI 按拓扑顺序逐个模块推进，每步验证，错误局限在单模块内。** 这正是架构设计带来的工程纪律。 |

### 能力 2：全链路自动验证 + 性能调优 + 崩溃自诊断

| 元素 | 内容 |
|------|------|
| **能力标题** | 质量闭环：集成测试 → 压测监控 → 性能调优 → 崩溃诊断，全程无人 |
| **价值描述** | 代码写完只是开始。AI 自动启动 MinIO + Ballista Scheduler + Executor 集群，跑 E2E 集成测试。然后一键压测，实时监控 CPU/内存。发现瓶颈自动调优。遇到崩溃自动诊断根因并修复。**这才是 AI 真正替代 80% 隐性工作的地方。** |
| **提示词示例** | "代码全部开发完了。启动完整分布式环境跑 E2E 测试。然后对 MapPartition 做性能压测，监控每个 executor 进程的 RSS 和 CPU，输出性能报告。压测过程中任何崩溃，自动分析并修复。" |
| **AI 自动执行** | ① 启动 MinIO + Scheduler + Executor → ② 注册表、导入测试数据 → ③ 跑 E2E 测试（SQL 查询 + 结果验证）→ ④ `bench.sh` 参数化压测（可配 executor 数 / 并发 / 数据量）→ ⑤ 逐进程 RSS/CPU 采集 → ⑥ 发现 IPC 序列化内存峰值 3× → **自动替换为 C Data Interface 零拷贝方案** → ⑦ 发现 `split_batch_by_key` O(n×k) → **自动优化为 O(n) HashMap** → ⑧ 发现 `dlopen` 重复加载 → **自动加 OnceLock 缓存** |
| **惊悚点：崩溃自诊断** | 压测中随机 crash（有时 10 分钟，有时 2 小时）。传统方式：gdb + valgrind + 翻 Arrow FFI 源码，半天起步。**AI 分析 crash 日志 → 直接定位到 `ManuallyDrop::into_inner` 和 `.so` 卸载后 release callback 的所有权竞争 → 3 个 FFI 生命周期 Bug 全部修复**（double-free、use-after-free、跨版本 ABI 兼容性验证）。人没有看一行 crash 日志。 |

### 能力 3：交付物自动生成 + 可复用资产沉淀

| 元素 | 内容 |
|------|------|
| **能力标题** | 资产沉淀：设计文档 + 性能报告 + API 文档 + SDK + 脚手架，自动生成 |
| **价值描述** | 项目开发完不是终点。AI 自动从代码和实验数据中提取信息，生成完整的设计文档、性能报告和使用手册。同时将开发过程中形成的模式沉淀为可复用资产（SDK crate + 脚手架脚本 + 压测工具），**下一次同类项目只需 5 秒初始化**。 |
| **提示词示例** | "项目开发完了。帮我输出三份文档：① DESIGN.md 架构设计文档 ② PERF.md 性能压测报告 ③ README.md 使用手册。然后把 processor 开发模式抽成 SDK crate 和 scaffold 脚手架脚本。" |
| **AI 自动执行** | ① 从代码结构 + commit 历史提取架构决策 → **DESIGN.md**（含 7 层管线图、C ABI 生命周期表、C Data Interface 所有权转移规则）→ ② 从 bench.sh 输出提取性能数据 → **PERF.md**（含不同 executor 数/并发的吞吐对比、内存峰值分析、优化前后对比）→ ③ 从公开 API 提取使用示例 → **README.md**（含快速开始、API 文档、参数说明）→ ④ 抽 SDK crate：`PartitionProcessor` trait + `export_partition_processor!` 宏 → ⑤ 写 `scaffold.sh`：`-p processor_name -c crate_name` 一键生成新 `.so` 项目（含 Cargo.toml 模板、processor 骨架、client 模板） |
| **可复用性** | 三份文档之间有交叉引用（README → DESIGN 了解架构、README → PERF 了解性能），构成完整的知识体系而非孤立文档。SDK + scaffold 让后续新增算子**从 2-3 天降至 5 秒**。 |

## 补充说明区

| | |
|---|---|
| **人机分工边界** | **人**：架构设计（7 层管线拓扑、C ABI 生命周期协议、C Data Interface 所有权模型）、领域约束（DistributeBy 分区语义）。**AI**：全部模块实现、测试编写与执行、集成验证、性能调优、崩溃诊断、文档生成、工具链构建。**边界清晰：人做不可替代的创造性决策，AI 做一切可自动化的执行。** |
| **方法论可迁移** | 该模式不限于 Ballista/Rust。任何需要多模块协同的系统级项目（微服务、数据管道、编译器前端），只要人能画出模块依赖图和接口契约，AI 就能按同样的"拓扑顺序开发 + 每步验证 + 全链路闭环"流程执行。 |
